# KHStrategyLab save1 전략 파트 개발 요약

작성 기준: `save1` 브랜치 / 조건01 후보 추적 및 매수신호 텔레그램 테스트 적용 후

---

## 1. 현재 전략 개발 위치

현재 전략 파트는 **실매수 전 테스트 단계**다.

실제 주문이나 가상매수는 아직 실행하지 않고, 조건이 충족되면 텔레그램으로 **매수신호 발생** 알림만 보내는 구조까지 연결했다.

```text
조건01 후보 저장
→ 기준봉 저장
→ KRX/NXT 시장구분 저장
→ 전략별 분봉/이평선 계산
→ 조건 충족 시 텔레그램 알림
→ 주문 없음
```

---

## 2. 조건검색 역할 정리

### 조건00

조건00은 화면 표시 전용이다.

```text
조건00
→ 화면 표시
→ 파일 저장 안 함
→ 가상매수 안 함
→ 전략 후보 아님
```

조건00 종목은 메인 화면 관제용으로만 사용한다.

### 조건01

조건01은 전략 후보 저장용이다.

```text
조건01
→ candidate_universe_active.json 저장
→ 6일 보관
→ 7일째 추적목록에서 삭제
→ 기준봉 저장 대상
→ 매수신호 테스트 대상
```

같은 종목이 오전, 오후, 다음날 다시 조건01에 들어와도 `FirstSeen`은 새로 갱신하지 않는다. 6일 보관 기간이 다시 늘어나지 않도록 했다.

---

## 3. 후보 저장 파일 기준

현재 후보 저장 기준은 `Storage/CandidateUniverse` 폴더다.

```text
Storage/CandidateUniverse/candidate_universe_active.json
→ 현재 6일 보관 중인 조건01 전체 추적목록

Storage/CandidateUniverse/candidate_universe_yyyyMMdd.json
→ 해당 날짜에 FirstSeen 된 신규 조건01 후보만 저장

Storage/CandidateUniverse/candidate_universe_expired_yyyyMMdd.json
→ 해당 날짜에 7일차로 삭제된 후보 기록
```

예전 `Storage/watch_candidates.json`은 더 이상 사용하지 않는다. 남아 있으면 삭제 대상이다.

---

## 4. 조건00/조건01 섞임 방지

키움 조건검색 응답에서 `seq`가 빠지는 경우가 있어, 이전에는 00번 응답 종목이 01번 후보로 섞일 가능성이 있었다.

현재는 응답 1건당 역할을 한 번만 판단하고, 종목 item마다 pending queue를 다시 소모하지 않도록 수정했다.

```text
CNSRREQ 응답 1건
→ response role 1회 확정
→ 모든 item은 같은 role로 처리
→ 00번 종목이 01번 후보로 섞이는 문제 방지
```

## 5. 기준봉 저장 방식

조건01 후보가 들어온 순간에는 기준봉 OHLC를 바로 저장하지 않는다.

```text
조건01 편입
→ Code / Name / LastPrice / FirstSeen 저장
→ KRX/NXT 시장구분 저장
→ 기준봉 OHLC는 일단 0
→ 장마감 후 기준봉 저장 타이머에서 OHLC 저장
```

기준봉 저장은 장마감 후 15:40 이후 1회 저장 흐름으로 동작한다. 새벽 07:00 전 재실행 시에는 전일 기준봉 저장 보완도 허용한다.

---

## 6. 기준봉 허리값 공식

처음에는 고가와 저가의 중간값을 검토했지만, 갭상승 봉에서 기준이 꼬일 수 있어 최종적으로 아래 공식으로 통일했다.

```text
BaseHalfPrice = (전일종가 + 기준봉 종가) / 2
```

이 값은 매수 가능 여부를 판단하는 핵심 안전장치다.

```text
현재가 또는 판단봉 종가 > BaseHalfPrice
→ 매수 판단 가능

BaseHalfPrice 이하
→ 매수 금지 또는 대기
```

## 7. KRX/NXT 시장구분 저장

조건01 검색식은 KRX/NXT를 고정해서 분리하는 것이 아니라, 후보가 들어온 뒤 해당 종목이 NXT 가능 종목인지 확인해서 저장한다.

후보 JSON에는 다음 필드가 저장된다.

```json
{
  "StrategyMarket": "KRX 또는 NXT",
  "ConditionMarket": "조건01",
  "MinuteChartMarket": "KRX 또는 NXT",
  "RealtimePriceMarket": "KRX 또는 NXT",
  "DisplayMarket": "KRX 또는 NXT",
  "NxtEnabled": true,
  "StrategyCode": "KRX_CONDITION01_PULLBACK_BREAK 또는 NXT_CONDITION01_PULLBACK_BREAK",
  "StrategyGroup": "KRX_CONDITION01 또는 NXT_CONDITION01",
  "MarketResolveSource": "ka10100",
  "MarketResolvedAt": "..."
}
```

KRX/NXT를 분리한 이유는 매수전략 자체가 크게 달라서가 아니다. 같은 종목이라도 KRX와 NXT의 분봉, 이평선, 고가 기준이 다를 수 있기 때문이다.

```text
전략 조건 = 일단 동일
분봉 데이터 / 이평선 숫자 = KRX/NXT 별도
```

## 8. 전략 등록 구조

현재는 같은 전략 조건으로 시작하지만, 나중에 반드시 달라질 수 있다고 보고 KRX/NXT 전략을 별도로 구성했다.

```text
KRX_CONDITION01_PULLBACK_BREAK
→ KRX 후보 전용
→ KRX 분봉 기준
→ KRX 이평선 숫자 사용

NXT_CONDITION01_PULLBACK_BREAK
→ NXT 후보 전용
→ NXT 분봉 기준
→ NXT 이평선 숫자 사용
```

즉, 전략은 “같지만 다르다”는 전제로 구성한다.

```text
공통 흐름은 공유
시장별 데이터 소스와 상태값은 분리
나중에 KRX/NXT 조건을 따로 수정 가능하게 유지
```

## 9. 현재 매수신호 조건

현재 테스트 전략의 기본 흐름은 아래와 같다.

```text
조건01 후보
→ 기준봉 저장 완료
→ 10분봉 60선 아래 이탈 확인
→ 10분봉 60선 재회복 확인
→ 기준봉 허리 위 유지
→ 10분봉 10선 눌림/접촉 확인
→ 5분봉 20개 신고가 돌파
→ 매수신호 발생
→ 텔레그램 알림
→ 주문 없음
```

각 단계는 바로 매수 조건이 아니라 상태 전환 조건으로 본다.

---

## 10. 단계별 추적 상태 개념

전략은 if문 한 번에 모든 조건을 몰아넣는 방식이 아니라, 상태값을 단계별로 넘기는 구조가 맞다.

```text
WAIT_BELOW_MA60
→ 기준봉 이후 10분봉 60선 아래 이탈 대기

BELOW_MA60_PULLBACK
→ 60선 아래 눌림 확인, 재회복 대기

RECOVERED_MA60
→ 60선 재회복 확인, 기준봉 허리 위와 10선 눌림 대기

TOUCHED_MA10_ABOVE_BASE
→ 10분봉 10선 눌림/접촉 확인

WAIT_5M_HIGH20_BREAK
→ 5분봉 20개 신고가 돌파 대기

BUY_SIGNAL
→ 매수신호 발생. 현재 테스트 단계에서는 텔레그램 알림만 전송
```

## 11. 이평선 계산 기준

이평선은 차트 전체를 매번 다시 그리는 방식이 아니라, 숫자 상태값으로 만들어가야 한다.

```text
10분봉 완성
→ 종가 추가
→ 최근 60개 종가 유지
→ 10분봉 MA60 갱신
→ 최근 10개 종가 유지
→ 10분봉 MA10 갱신
→ 상태 전환 체크
```

5분봉은 매수 트리거용이다.

```text
완성된 최근 5분봉 20개 고가 중 최고값
→ FiveMinHigh20
→ 현재가 또는 최근 종가가 이 값 돌파 시 매수신호 후보
```

현재 진행 중인 5분봉의 고가는 돌파 기준에 포함하지 않는 것이 안전하다.

---

## 12. KRX/NXT 이평선 분리 기준

KRX 후보는 KRX 분봉으로 이평선을 만든다.

```text
KRX 후보
→ KRX 10분봉 MA60
→ KRX 10분봉 MA10
→ KRX 5분봉 High20
→ KRX 현재가 기준
```

NXT 후보는 NXT 분봉으로 이평선을 만든다.

```text
NXT 후보
→ NXT 10분봉 MA60
→ NXT 10분봉 MA10
→ NXT 5분봉 High20
→ NXT 현재가 기준
```

NXT 분봉 조회는 실제 키움 응답을 보며 확정해야 한다. 현재 구조는 NXT 후보에 대해 `_NX`, `_AL`, 6자리 코드 fallback 가능성을 열어둔 테스트 단계다.

---

## 13. 텔레그램 알림 연결

현재 테스트 단계에서는 매수 조건이 충족되면 텔레그램으로 알림만 보낸다.

실제 매수 주문이나 가상매수는 실행하지 않는다.

알림 메시지에는 시장과 전략코드가 들어가야 한다.

```text
🔔 [KHStrategyLab] KRX 매수신호 발생
종목명(코드)
전략: KRX_CONDITION01_PULLBACK_BREAK
분봉기준: KRX
현재가: ...
5분20고가: ...
10분MA60: ...
10분MA10: ...
기준봉 허리: ...
※ 테스트 단계라 매수 주문은 실행하지 않음
```

NXT 후보는 다음처럼 구분된다.

```text
🔔 [KHStrategyLab] NXT 매수신호 발생
전략: NXT_CONDITION01_PULLBACK_BREAK
분봉기준: NXT
※ 테스트 단계라 매수 주문은 실행하지 않음
```

## 14. 현재 전략 파트 상태 요약

완료된 부분은 다음과 같다.

```text
조건01 후보 저장
6일 보관 / 7일 삭제
조건00/조건01 역할 분리
조건01 날짜별 신규 후보 저장
기준봉 장마감 저장
BaseHalfPrice 공식 통일
KRX/NXT 시장구분 저장
KRX/NXT 전략코드 분리
매수신호 조건 초안 구현
텔레그램 알림 연결
실매수 없음
```

아직 보완이 필요한 부분은 다음과 같다.

```text
KRX/NXT 분봉 조회 방식 실전 확인
이평선 숫자 캐시 고도화
후보 수 증가 시 분봉 조회 캐시 적용
상태값 파일 저장/복원 세분화
중복 알림 기준 정교화
가상매수 연결
실주문 연결
```

## 15. 다음 테스트 계획

내일부터 테스트할 때 확인할 항목은 다음과 같다.

```text
1. 조건01 후보가 들어오는지
2. 후보 JSON에 StrategyMarket / NxtEnabled / StrategyCode가 정상 저장되는지
3. 장마감 후 기준봉 OHLC / BaseHalfPrice가 정상 저장되는지
4. KRX 후보와 NXT 후보가 다른 전략코드로 분리되는지
5. 분봉 조회가 정상 응답하는지
6. 10분봉 MA60 / MA10 / 5분봉 High20 계산이 이상 없는지
7. 조건 충족 시 텔레그램 알림만 발생하는지
8. 주문이 실행되지 않는지
```

## 16. 현재 최종 기준

전략 엔진은 무겁게 만들지 않는다.

```text
화면 전체 차트 재계산 금지
후보 전체 분봉 장기 저장 금지
실시간 틱마다 이평선 재계산 금지
조건01 후보만 평가
10분봉/5분봉 완성 기준으로 숫자 갱신
매수신호는 텔레그램 알림으로 먼저 검증
```

최종 방향은 다음과 같다.

```text
전략 조건은 단순하게
시장별 데이터는 분리해서
상태값은 단계별로 저장하고
실제 주문은 충분히 검증한 뒤 연결한다
```
